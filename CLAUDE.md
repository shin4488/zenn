# CLAUDE.md

ZennのGitHub連携で記事を管理するリポジトリ。mainブランチへのpushで自動デプロイされる。

- 公開アカウント: https://zenn.dev/shin4488
- 既存記事(Webエディタ作成分)は自動生成スラッグ(16進数)なので、このリポジトリで使う英単語ベースのスラッグとは衝突しない。新規記事のスラッグが既存記事と重複しないことだけ確認すること(重複すると既存記事が上書きされる)

## 構成

- `articles/` … 記事のMarkdown。ファイル名がそのままスラッグ(URL)になる
- `images/` … 記事に貼る画像。記事スラッグごとにサブディレクトリを切る(`images/<記事スラッグ>/xxx.png`)
- `assets/svg/` … 図の元データSVG。Zennのデプロイ対象外。記事スラッグごとにサブディレクトリを切る
- `books/` … Zennの本(未使用)

## Zennの制約(違反するとデプロイエラーや表示崩れになる)

- 記事のスラッグ(ファイル名)は半角英数字・ハイフン・アンダースコアで12〜50字
- 本文の画像参照は `/images/<記事スラッグ>/xxx.png` の**絶対パスのみ**。相対パス(`./images/...`)は不可
- `images/` に置けるのは png / jpg / jpeg / gif / webp のみ(**SVG不可**)、1ファイル3MB以内
- 公開状態はフロントマターの `published: true/false` で制御。`published_at` で予約公開も可能
- 記事の削除はファイル削除だけでは完了しない(Zennダッシュボード側の削除も必要。逆にファイルが残っていると復活する)

## 図の作成・修正フロー

`assets/svg/` のSVGを編集し、Chromeヘッドレスで `images/` にPNGとして書き出す。

```sh
CHROME="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
"$CHROME" --headless=new --disable-gpu \
  --screenshot=images/<記事スラッグ>/figN.png --window-size=<幅>,<高さ> \
  "file://$PWD/assets/svg/<記事スラッグ>/figN.svg"
```

window-size は各SVGの width / height 属性(viewBoxの2倍で書き出している)に合わせること。

## 執筆スタイル

- です・ます調で、人間が書いた自然な語り口にする。実際に指摘を受けたAIっぽい表現の例:
  - 読者への定型的な気遣い・保証:「〜という人でも大丈夫なはずです」
  - 体言止めの三連列挙と雑な締め:「レーザー、剣の軌跡、弾丸。だいたいこのどれかです」
  - 抽象的に構えた言い回し:「難しさは〜に集中しています」「今回の主役、内積と外積」
  - 比喩ラベルの多用:「最初の道具は三角関数です」「道具が揃ったので」
  - 蛇足の一言:「地味にハマります」
  - 馴れ合い・したり顔の語りかけ:「そうはいかないんですよね」
  - 概念や図形の擬人化:「内積の分類が〜と見抜く」「比較相手」「線分を相手にする」「円の中心が〜にいる」(図形の位置は「いる」ではなく「ある」で書く)
  - 定型の締め(「いかがでしたか」)、箇条書きの乱発、「まず・次に・最後に」の機械的な繰り返し
- 構成は「理論の説明 → 実装コード」の順。図とコードを織り交ぜ、文章だけの長い区間を作らない
- 理論を説明するときは、なぜその理論を説明するのか・最終的にゴールのどこで使うのかを、説明の前に書く(記事の冒頭でゴールを提示し、各セクションの導入でゴールとのつながりを示す)
- 記事は読了5〜7分程度に収める
- 掲載するコードは必ず型チェック(`tsc --strict`)と実行で動作確認してから載せる
